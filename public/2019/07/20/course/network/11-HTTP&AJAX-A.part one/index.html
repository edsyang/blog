1. 本地存储

  1. 锤子手机(https://www.smartisan.com/)模态框：第一次打开页面时，弹出了模态框，点击遮罩层会消失，并且再次刷新页面时，这个模态框不再出现，这就说明它已经记录了用户已经关闭过模态框以及关闭的时间
  2. 京东(https://www.jd.com/)购物车：不登录也可以将物品添加到购物车，而没有登录时为匿名，京东不知道用户到底是谁，所以不可能把数据存在服务器上；
  3. 一个网站有n多页面，只要在其中一个登录，其它页面就可以拿到登录信息

    以上问题都是用本地存储技术解决：

  4. 在浏览器中每次打开一个页面都会形成一个新的、独立的全局作用域；所以在浏览器打开两个页面，一个A、另一个B，因为A、B都会形成不同的作用域，所以A、B不能互相访问变量；但是实际项目中又多有这样的需求，比如在A中登录了，B也可以拿到登录信息
  5. 页面都是在浏览器中打开的，所以可以让浏览器充当中介，让A页面把信息存储到浏览器，B页面需要的时候就去浏览器上取，这个操作就是本地存储

    常见的本地存储：

  6. cookie（http协议的一部分）
  7. localStorage 本地存储（HTML5新增，浏览器技术）
  8. sessionStorage 会话存储（HTML5新增，浏览器技术）

    localstorage和cookie的区别：

  9. ls是H5新增的浏览器本地存储，是浏览器技术；cookie属于http协议的一部分；ls和cookie都是存在浏览器的
  10. ls不兼容IE低版本的，cookie兼容IE低版本；
  11. cookie会随着http协议传递，大小有限（能存的东西很少），ls存储的空间较大；尽量不要向cookie中存东西，会占用http的带宽
  12. cookie可以设置过期时间，ls不能设置；如果要求ls有过期时间，设置一个定时器每隔一秒看一下ls中的东西是否过期，如果过期就删除ls中存储的内容（一旦清空ls后要把定时器清除掉）

    localStorage和sessionStorage的区别： -> localStorage、sessionStorage是HTML5新增的API，IE低版本不能用

  13. localStorage（简写ls）是永久存储（如果用户/我们删除了就没有了）；sessionStorage是会话存储，sessionStorage中存储的数据生命周期就是整个会话，如果会话关闭数据就也没有了，只在当前会话有效不能跨会话使用
  14. （同源/域（要求协议、域名、端口号相同）策略下）打开两个页卡，如果用localStorage，在A页卡中设置，在B页卡中可以获取；但在sessionStorage中不行

1.1 localStorage

本地永久存储（除开发者使用代码触发删除或者用户手动清除）

localStorage是挂载在window上的对象
    localStorage.setItem(key, value) 向localStorage中添加数据
    - key：名 ；value：值 ；key和value要求是字符串或者数字，如果不是就会默认转为字符串
    - 注意：不能设置两个同名的属性，如果同名，后面的会覆盖前面的

    localStorage.getItem(key) 根据key获取localStorage中存储的数据
    - 获取localStorage中不存在的key会返回null

    localStorage.removeItem(key) 删除指定key的数据
    localStorage.clear() 清空localStorage，一般退出登录时需要清空，将localStorage中的东西都删除

在哪里看localStorage：打开控制台 -> Application -> Storage -> localStorage

！！！真实项目中经常会把对象或者数组存储到localStorage中的需求，就需要把对象或者数组变成JSON格式的字符串 JSON.stringify()再存, 等使用时记得JSON.parse()变成对象

1.2 sessionStorage

  • sessionStorage：客户端本地存储的一种，会话存储
  • session：服务端技术，存在服务器上

注意：会话存储；刷新并不会使sessionStorage失效，但是关闭页面后sessionStorage中的数据就没有了

打开一个页卡就是和服务器形成了一个新的会话；sessionStorage只在当前会话中有效，不能在其他会话中使用另外会话中的数据；

setItem(key, value) 存储，key和value是字符串类型的 
getItem(key) 获取指定key的值
removeItem(key) 删除指定key的数据

Unix 时间戳是以秒为单位的，js/浏览器的时间戳是以毫秒为单位的，PHP的时间戳就是unix时间戳；注意转换！如果服务端让你传给它时间戳，要问一下是毫秒还是秒。如果服务端要秒，要用js时间戳除以1000；如果服务端返回给你的是unix的时间戳，需要乘1000转换js时间戳【unix时间戳比js少三位，一看就比较短】

2. 发布网站

1. 项目：前端+后端代码

2. 域名：可以去万网（https://wanwang.aliyun.com/）购买；每个域名都是不可重复的，域名的作用在于用户记住、访问网站；在没有域名时需要使用IP，IP是一串数字的组合，如百度61.135.169.125:443

3. DNS服务：Domain-Name-System DNS由DNS服务器提供，它负责把域名解析成IP地址

4. 服务器：就是一台高性能的电脑，服务器和普通电脑不同在于服务器追求稳定，不能总出问题；项目的代码要想给用户提供服务，需要把代码部署到服务器上；每个服务器都有一个IP地址，并且靠这个IP地址连接网络给外界提供服务

5. ftp上传：File Transfer Protocol 把代码和其他文件上传到服务器上（有很多ftp工具，如FileZilla、IDE也带、写脚本也可以上传）

6. 端口：服务器不仅可以提供一个服务，一个服务器可以分为若干的小房间，一个小房间对应一个编号，每个小房间都可以提供一个服务；编号的取值范围是：0 - 65535；一般不会都开放，一般只开常用的；http协议默认使用80端口，https协议默认使用443端口，数据库默认使用3306；

7. 搜索爬虫：做搜索的公司有一个程序，每隔一个间隔都去各大网站把热词都拿回来放到自己的搜索库里面；此外还收集网站上的关键词：TDK也放到搜索库中。当进入到搜索东西时，它会把我们输入的关键词和它搜索库中的东西比较，把它觉得相似的给我们；

webstorm或者vscode-liverServer可以在本机启动一个服务，所以本机充当了服务器的角色，在我的电脑上访问localhost: 63342/xxx/xx，此时我的电脑又充当客户端的角色，所以此时我的电脑既是客户端又是服务端，此时不需要互联网

3. HTTP协议

前后端交互（传输数据）时要遵守的规范

HTTP

HyperText Transfer Protocol 超文本传输协议；用于客户端和服务端通信的一种协议，在客户端和服务端之间传输文本、图片、音视频等内容；是客户端和服务端之间约定的规范，双方按照规范传递和接收数据

HTTPS

加密的传输协议，在数据传递的时候更加安全；在协议中增加了密钥，而且是建立连接的客户端和服务端才知道；用HTTPS的请求不容易被篡改；一般对安全有要求的网站都需要用HTTPS，如银行、支付宝、微信、QQ等都需要使用HTTPS

HTTP协议负责在客户端和服务端传递数据；把客户端想要的东西告诉服务器（这个过程叫请求request），服务器会把客户端想要的东西通过http协议返回给客户端（这个过程叫响应response）

  • HTTP协议通信分为两个阶段：请求阶段request，响应阶段response

3.1 ⚠️前后端交互模型⚠️

在浏览器的地址栏输入一个url，按下enter键，到出现页面经历的事情：

  • 请求阶段request
    • 1.获取浏览器地址栏中用户输入的地址，（not important：先在本地的DNS缓存和host文件，这两个里面存储的也是ip和url的对应关系）然后把地址发送给DNS服务器
    • 2.因为在网络中通信使用的都是IP地址，DNS服务器把用户输入的url解析成IP地址（每个服务器都有一个IP地址）
    • 3.把请求的内容（数据）发送到DNS服务器解析出来的IP地址对应的服务器
  • 响应阶段response（默认http找80端口，https找443端口；可修改）
    • 4.服务器接收到客户端的请求后，分析请求内容，整理客户端需要的内容（可以是数据还可以是html、css、js、img等静态文件），然后把响应的内容通过http协议返回给客户端
    • 5.服务端的响应中会告诉客户端响应内容的类型；在响应头中有一个Content-Type的属性，表示响应内容的类型，目的是告诉客户端按照不同的类型去解析返回的响应内容
  • 浏览器的渲染阶段render
    • 1.解析html文档，根据html节点间关系组织（绘制）成DOM tree；解析HTML时遇到link、img、script、audio、video等标签会再次发送http请求，去请求对应的资源（link的href、script和img的src属性指向的静态资源文件）
    • 2.解析CSS文档，形成CSSOM tree
    • 3.把DOM tree和CSSOM tree组合起来形成渲染树（Render tree）
    • 4.把Render tree交给GPU绘制页面

      script标签中有一个async属性，表示不用等js加载完 <script async></script>；link的href，img、script、audio、video的src都会让浏览器自动发起http请求，这些请求都是GET请求

DNS解析：所有连接在互联网里面的设备都有一个IP地址；但是人类更擅长记忆url，如 www.baidu.com；DNS解析就是负责把url解析成IP地址

前端做性能优化专门有一个从http层面优化的方向

3.2 ⚠️http的三次握手和四次挥手

HTTP协议在客户端和服务端建立连接和断开连接不是一次就完成的，而是要经历三次握手和四次挥手；三次握手和四次挥手指建立和断开连接的时刻，不涵盖具体传递数据的细节

SYN（synchronous建立连接）；ACK（acknowledgement 确认、响应）；FIN（finish结束连接）；SYN/ACK/FIN不是客户端和服务端要传递的数据，就是用来测试连接成功与否的东西；访问客户端真正要传递给服务器的数据时，请求首页；服务器真正响应给客户端的也是网站首页的HTML文件；

  • 三次握手（建立连接）
    • 第一次握手：客户端向服务端发送SYN码数据包，表示客户端请求和服务端建立连接
    • 第二次握手：服务端收到客户端的连接请求后，会发送ACK数据包给客户端给客户端，表示收到连接请求，询问客户端是否真的需要建立连接
    • 第三次握手：客户端收到ACK码之后会检验是否正确，如果正确，客户端会再次发送ACK码给服务端，表示确认建立连接

      三次握手都成功以后才会建立连接，然后才会发送数据

  • 四次挥手（断开连接）
    • 第一次挥手：当客户端发送请求数据结束时，会发送FIN码数据包给服务端，表示告知服务端，客户端要给服务器的数据传输完了
    • 第二次挥手：当服务端收到FIN后，会发送ACK给客户端，表示服务端已经知道客户端传递完了；客户端收到ACK以后就会把传递数据给服务端的通道关闭
    • 第三次挥手：当服务端把响应的数据（传输给客户端的数据）发送完毕时，会发送一个FIN给客户端，告知客户端响应的数据已经发送完毕
    • 第四次挥手：当客户端收到FIN后，会发送ACK码数据包给服务端，告知服务端，客户端已经知道服务端数据发送完毕；服务端收到ACK码后把数据传递（输）通道关闭

5. HTTP报文

HTTP报文是前后端通过http协议交互/通信时用于承载交互信息的载体

报文结构分为三段：

  • 报文首部
  • 空行（CR+LF）：Carriage Return 回车符 + Line Feeding （CR+LF组成一个空行）
  • 报文主体

因为前后端交互分为request、response两个阶段，所以对应的报文分为两种：

  • 请求报文：
    • 报文首部：请求行【请求方式（GET、POST…）、http协议版本、URL】和请求头（通用、实体、其他）
    • 空行（CR+LF）
    • 请求体（POST请求才有请求体，GET请求浏览器会自动忽略）
  • 响应报文：
    • 报文首部：状态行【http协议版本、响应状态码（http status）、响应码对应短语】和响应头
    • 空行（CR+LF）
    • 响应体：服务端响应的主要内容

      请求报文和响应报文结构相同，但里面的内容不一样

6. AJAX

AJAX缺点：不能跨域，如果跨域需要做特殊处理

6.1 AJAX基础

AJAX：Asynchronous JavaScript And XML 异步的JavaScript和xml；是一种浏览器技术。

  • XML (Extensible Markup Language)：可扩展标记语言，像html一样有层级，并且可以自定义标签；为了让数据有结构，早年间使用的都是XML
  • JSON (JavaScript Object Notation)：是一种数据交换格式；有JSON格式的对象，JSON格式的字符串
    • JSON.parse() 把JSON格式的字符串转成对象
    • JSON.stringify() 把对象转成JSON格式的字符串
    • JSON.parse(JSON.stringify(对象)) 可以深克隆对象

      现在绝大多数公司都用JSON，JSON也可以清晰的表示数据结构

XML 和 JSON 两者的差别

  1. xml是重量级的，json是轻量级的
  2. xml在传输过程中比较占带宽，json占带宽少，易于压缩
  3. xml和json都用在项目交互下，xml多用于做配置文件，json用于数据交互
  4. json可用jackson，gson等方法解析(易于解析)，xml可用dom，sax，demo4j等方式解析(解析会花费较多的资源和时间)

6.1.1 服务端渲染和全局刷新

在没有AJAX的时候，数据都是由服务（后）端绑定的；前端只负责把设计稿还原成html，然后就把html交给后端；后端通过服务端技术把数据从数据库中取出来，然后直接把数据绑定在html中；当请求页面时，拿到的页面都是已经由后端绑定好数据的页面（服务端绑定数据，这种模式叫做服务端渲染SSR（Server-Side-Rendering））

  • 服务端渲染SSR
    • 优点：用户看到的页面都是带有数据的，不需要等；
    • 缺点：如果这个页面中只有一部分数据需要更新，但是必须刷新整个页面才能获取最新的数据（因为数据都是服务器绑定的，要想获得最新的数据，就要让服务端再绑定一遍）；这种获取数据的方式称为全局刷新；全局刷新体验不好

有了AJAX以后，发送AJAX请求去服务端请求数据，然后拿到数据后通过前端技术把数据绑定到页面中（动态创建DOM+appendChild、拼接字符串）；这种操作不需要刷新整个页面就能更新页面数据，这种方式称为局部刷新

6.1.2 使用AJAX

  1. 创建AJAX实例
  2. 调用open方法
  • open参数：
    • 2.1 http-method 请求方式（请求方法）
    • 2.2 接口
    • 2.3 是否异步；true异步，false同步
  1. 监听xhr的onreadystatechange事件
  • readyState：表示当前AJAX实例所处状态的一个数字，每一个数字代表不同状态
    • 0：创建一个AJAX实例成功
    • 1：已经调用过open方法，打开了路径
    • 2：表示客户端已经接收到服务的响应头（如果只要响应头可以在readyState为2时获取就行）
    • 3：表示客户端正在接收响应体
    • 4：表示响应体中的数据已经全部接收完毕

      异步的时候，readyState发生变化会触发readystatechange事件；所以每次变化都会触发一次，但只有等于4时才有意义；同步的时候只有为4时才触发

  1. 发送AJAX请求
  • send是可以传参的，传递的参数会写到请求体中

6.2 http-method

常用的请求方式

在AJAX中，http请求方式就是open() 方法的第一个参数；

接口都是服务端定出来的，服务端会给一个接口文档；接口文档中会对请求方法、传递的参数以及返回值的结构和类型 https://testsuper.luxiaoquan.com/swagger#!/Route/post_scenic_scenicId_route（RESTful API）

示例：

  • 接口：/xxxx/xxxx/xxxx 【第一个/表示的是本网站】
  • 功能：功能介绍
  • 请求方式：http method

1. GET：一般用于从服务器获取数据，但也可以向服务器传递/发送少量的数据

GET请求的特点：

- 1. GET请求通过url末尾以问号传参的形式向服务器传递数据，如?key1=value1&key2=value2（通过location.search 获取地址栏里面url的上的问号传参）；
      1.1 向url末尾拼接参数（这些参数也叫查询字符串），如果原有的url有?就不需要再写?，直接用&符和之前的分隔
      1.2 这些参数以key=value的形式存在，key一般都是服务端定好的，服务端要求传什么就传什么
- 2. 因为GET发送数据给服务端，都是把数据拼接在url末尾，但是url的总长度是有限的；一般IE是2k，Chrome是8k；所以GET请求能传递/发送给服务器的数据量很小
- 3. GET请求传递数据是以明文的形式传递，不安全
- 4. GET请求容易走缓存，一般情况下访问接口的数据不需要缓存；所以一般情况下在url末尾拼接随机数或者时间戳来清除缓存（原理：浏览器检测到url发生变化，就不会用缓存的内容，而是重新请求数据）

GET 请求的参数 在控制台的headers最下面 QueryStringParameters

2. POST：一般用于把数据传递给服务器或需要传递大量数据

POST请求的特点：

- 1. post请求把数据放到请求体中传递给服务端（字符串类型的）
- 2. 发送给服务器的数据是放到请求体中的，所以传递数据的大小没有限制
- 3. post请求是把数据放到请求体里面，相对来说安全一点
- 4. post请求不会走缓存

POST请求的参数 在控制台的headers最下面 RequestPayload

其他请求方式：

  • PUT：修改某些内容
  • DELETE：从服务器上删除数据
  • HEAD：只获取响应头，不需要响应体
  • OPTIONS：查询服务器支持的HTTP请求方式
  • TRACE：回显请求的内容，一般用于测试
  • PATCH：对PUT的补充，用于修改某些内容

RESTful API（需要服务器架构支持RESTful，前端才能使用RESTful的接口）；以HTTP-METHOD（get/post/put/delete…）为动词，表示请求的目的是什么，服务端提供统一的接口，根据不同动词采取不同的操作，返回不同的数据；

  • 接口：/order/:id（id时一个数字），如/order/123
    • GET /order/123 获取订单id为123的信息
    • DELETE /order/123 删除订单id为123的信息

6.3 http-status

xhr.status 是 http-status-code，表示http请求的结果状态；所有的http-status-code都是三位数字，表示http请求的结果如何，每个code还有一句对应的短语

控制台 -> Network -> 文件（Name） -> Headers

常用的http-status-code

2xx：表示成功

  • 200 OK：表示所有的东西都正常
  • 200 No Content：表示请求成功，但是服务端没有内容给客户端

    3xx：表示重定向

  • 301 Moved Permanently：永久重定向，一个域名被永久的指向了一个其他网站；（当访问一个永久重定向网址的时候，会收到一个301的响应状态码，浏览器检测到301以后，浏览器会从响应头中找到location的属性对应的url，然后浏览器会重新打开这个url）
  • 302 Moved Temporarily：临时重定向
  • 304 Not Modified（不修改）：走缓存（服务端觉得客户端之前请求过这个东西，而且服务器上的那一份没发生变化，告诉客户端用缓存就行）

    4xx：表示客户端错误

  • 400 Bad Request：参数传递不当导致的错误
  • 401 Unauthorized：权限不够
  • 403 Forbidden：服务端已经理解请求，但是拒绝响应
  • 404 Not Found 客户端请求的资源或数据不存在（上班后发现请求接口404，有两种情况：1.自己写错接口；2.服务端还没部署）

    5xx：表示服务端错误（遇到以5到头的错误，去找服务端） -> 这些状态码都是服务端设置的，如果出了问题找服务端

  • 500 Internal Server Error：服务器内部错误
  • 502 Bad Gateway：网关错误

6.4 AJAX同异步

  • 同步：阻塞的，前一个执行不完，后一个不能开始
  • 异步：非阻塞的，后面的执行不用等前面的执行完（异步任务放到等待任务队列中，等待任务队列中的任务谁先达到执行条件谁先执行）

    真实项目中用异步的偏多；但当这些接口互相依赖时，代码组织起来会比较麻烦，如有两个接口，第二个接口依赖于第一个接口

    用异步的方式请求两个接口，首先请求aside.json，等上一个结束后再请求banner.json

7. ⚠️同源策略（Same-origin Policy）⚠️

AJAX虽然好用，但是不能跨域；如果一定要跨域就需要做特殊处理

AJAX受同源策略的约束：

  • 同源策略：为了保证浏览器的信息安全，当前源中的信息只能在当前的源中使用；其他源如果想使用当前源中的信息，需要做特殊的处理
  • 同源策略要求通信的源的协议、域名、端口号都相同，如果三者中有一个不同就是非同源。非同源的通信叫做跨域，需要做特殊的处理；
    • http://www.baidu.com & https://www.baidu.com/abc/def （协议不同）
    • localhost:5500 & file://c://js/index.js （协议不同，fill://为file协议）
    • https://www.baidu.com & https://www.vip.baidu.com （域名不同）
    • localhost:63342 & localhost:5500（端口号不同）

常用的跨域解决方案：

  1. JSONP：利用script的src属性不受同源策略的约束，用script请求第三方接口，但是需要做一些特殊处理
  2. 服务端转发：只有客户端受同源策略的约束，服务端是不受其约束的；请求客户端的借口，然后让服务端在服务器上去请求第三方的接口，拿到第三方接口的数据返回给前端即可
  3. nginx转发：nginx是服务端应用程序，在服务器可以配置某种规则，nginx就会帮客户端转发请求到第三方（需要服务端和运维的支持）
  4. CORS：Cross-Origin-Resource-Sharing 跨域资源共享；目标源（域）需要设置一个Access-Control-Allow-Origin响应头；这个头可以设置成具体的源，也可以设置为*（一般不会设置为 *，只有 css、js、image等可能会用 *），表示任意的源都能访问；这种跨于解决方案，需要目标域服务端配合才能实现

7.1 JSONP

一种跨域技术、解决跨域问题的方式（和JSON没有直接关系）

JSONP的原理：利用了script的src属性不受同源策略的约束（link的href、img的src都不受约束），所以用script的src请求第三方的接口，但是需要做一点特殊处理

  1. 首先在页面中写一个script标签，src属性值是第三方的接口
  2. 声明一个函数fn，fn要有一个形参，把使用第三方数据的操作都写在fn里面
  3. 给前面的script的src上的接口多拼接一个callback=fn的参数
  4. 如果第三方的接口支持JSONP，会返回一个形如fn({xxxxx})的东西；script是用来加载JS代码的，script就认为fn({xxxxx})是JS代码，浏览器就会执行它；而fn()就是让fn这个函数执行，小括号内的东西就成为执行fn时的实参

    JSONP只能用来发GET请求；

callback不是死的，而是第三方接口定义好的，如果使用是需查阅第三方的文档
