项目开发流程

  1. 项目立项：宣布要做一个什么功能，讨论一下这个功能的可行性或者实现成本（严谨一点：可行性研究报告）
  2. 需求评审（如果项目不大，需求评审和项目立项一起完成）：由需求方（一般是产品经理）发起，组织设计的前端（FE）、后端（RD、BE）、测试（QA）、设计师（UI产品设计、UE用户体验）等开会。需求评审主要是讲述需求，讲述需求时还会解答大家对需求的疑惑；需求评审过后许产出无异议的产品需求文档（PRD）
  3. 需求确定后，前后端的开发开始技术调研，并且设计技术方案；在这个阶段如果用到了新技术，如地图、图表等，需要去学习调研新技术；而后端在做数据库表、接口的设计；设计师在制作设计稿
  4. 设计稿评审：由产品发起，组织 FE、UI、UE、QA 参与，讨论设计稿的实现难度以及验收标准；如果前端觉得这个效果实现起来很困难可以和设计师讨论（比方说‘如果要按照这样的实际会需要很长时间’）（如果项目小或者没有这样的规定，一般就是 FE 和 UI 的碰头会），设计稿评审产出的是设计稿（toB 的产品一般没有设计稿）
  5. 接口文档评审：由后端技术负责人发起，FE、QA 参与；目的是确定前后端交互的接口数量、前端传递的参数、接口返回值的数据结构、类型是否满足需求、其易用性如何；这一阶段，前端需要重点关注接口返回的数据是不是够，数据结构的易用性（在这个阶段如果有些接口过于复杂可以要求后端更改，拒绝更改问其理由，一向如此便是对么？），接口文档评审后会产出接口文档
  6. 排期：设计稿和接口文档都已经确认后，这个阶段给出的排期是一个精确地排期；没有意外情况，这个排期不能随意更改（如果需求变更、设计稿变更，如果变更特别大，或者提出的变更时间比较晚，需要考虑是否重新排期）；预估时间要乘二（因为还有太多外界影响的因素）
  7. 开发：排期结束之后进入开发阶段；如果需求比较紧急，每天早晚都会有站会（站着开个会）沟通进度
  8. 自测 + 联调：因为开发的时候，前端用的是接口文档以及接口返回的假数据开发；而后端也看不到前端的界面，所以在最后双方都开发完以后进行联合调试；联调就是把前端和后端的代码都上线到一台开发服务器上，通过开发服务器访问已经开发的应用，然后测试主要流程。如果公司比较严谨，QA 会提供 P1 或者 P0 级别的 CASE（测试用例：是一种文档，上面描述了如何操作以及该操作应该出现的结果；如果操作过后没有出现对应的结果就是写的代码有 bug（不一定是谁的，需要查）），在联调阶段执行 P1 或者 P0 的 CASE 进行自测，这个过程成为冒烟测试
  9. 提测：通过自测和联调后的代码提交给 QA，他们去验收功能，随着验收，如果出现 bug，要改
  10. 灰度发布：把应用上线到和线上服务器环境很相似的一台服务器上，然后再走一遍测试流程，如果出现 bug 开发需要优先处理
  11. 上线（发版）：把经过测试后的代码上线到为用户提供服务的服务器上，这个过程中需求方 和 QA 还会做线上验收；经过这一步后，外界的用户才能使用开发的功能
  12. 复盘会：论功行赏，不求有功但求无过
  • toB（to business 面向业务，如管理后台）
  • toC（to customer 面向用户，如 APP 等）

产品经理只是一个岗位，并不是管理岗
设计稿评审和接口文档评审不分先后，谁先完成谁先评审
上班有些事情一定要说出来，不说的话没人替你说出来
被开除不可怕，可怕的是找不到工作

GIT 分支开发流程

不包含回滚的操作

  1. 如果是大型项目一般是由项目负责人（一般是后端）开一个公共的开发分支（远程分支，本地分支需要自己开），例如 dev_0829；如果项目较小，自己开分支即可
  2. 如果多人协作的项目，不要直接在公共开发分支上开发；而是基于公共开发分支再切一个个人开发分支，例如 dev_0829_mars
  3. 开辟自己的开发分支后，在自己的分支上 add、commit
  4. 等到开发结束后（也可以写一点自测的没问题就合并一点），把自己的开发分支上的代码合并到公共开发分支上，即：将 dev_0829_mars 合并到 dev_0829
  5. 合并自己的分支到公共开发分支步骤
  • 1).将本地分支切换到公共开发分支上：git checkout dev_0829
  • 2).同步远程公共开发分支上的代码：git pull origin dev_0829
  • 3).把自己的开发分支合并到公共开发分支上：git merge dev_0829_mars --no-ff，esc，:q 退出即可
  • 4).如果有冲突，需要解决冲突；解决冲突后再 add、commit；然后 push 到远程公共分支上：git push origin dev_0829
  1. 如果开发完成后要提测，需要将 master 上的代码同步到公共开发分支上：master -> dev_0829
  • 1).本地切换到 master 分支上：git checkout master
  • 2).同步远程 master 代码到本地 master：git pull origin master
  • 3).切换到开发分支上：git checkout dev_0829
  • 4).把 master 上的代码合并到开发分支：git merge master --no-ff
  • 5).如果合并出现冲突，解决冲突，然后 add、commit；最后把代码 push 到远程开发分支上：git push origin dev_0829
  • 6).此时提测，把分支发给 QA 或者写在提测文档上
  1. 如果在测试阶段出现 bug，我们需要修改 bug，（可以）直接在公共的开发分支上修改，然后 add、commit、pull、push
  2. 上线：从提测到上线一般会间隔一段时间，上线前还需要在此同步 master 的代码到分支上；重复 6.1 - 6.5 的操作；经过这一步之后去 gitlab 上提交 merge request
  3. 提交完 mr（merge request）以后找有权限的人合并；合并完之后执行上线流程

JS中常见的几种报错类型

1、SyntaxError（语法错误）

解析代码时发生的语法错误

2、ReferenceError（引用错误）

3、RangeError（范围错误）

4、TypeError（类型错误）

变量或参数不是预期类型，比如，对字符串、布尔值、数值等原始类型的值使用new命令，就会抛出这种错误，因为new命令的参数应该是一个构造函数。

调用对象不存在的方法

5、URLError（URL错误）

与url相关函数参数不正确，主要是encodeURI()、decodeURI()、encodeURIComponent()、decodeURIComponent()、escape()和unescape()这六个函数。

6、EvalError（eval错误）

eval函数没有被正确执行

扫码登录

  • 以微信扫描网易云笔记为例
  1. 首先生成一张二维码，二维码就是一个 url 这个，这个 url 是微信服务器上的用户扫码登录的接口，url中有一个 app-id
  2. 生成二维码以后，在网页中开启一个定时器，每隔一定时间就去请求一个接口，询问扫描状态和登录状态
  3. 使用 微信 扫描二维码，就是通过微信内置的浏览器请求这个url，请求这个url时会把 app-id 传递给微信服务器，微信服务器收到以后校验 app-id 是否有效
  4. 在此之前，需要网易家的后端接通微信后端的的二维码登录，如果 app-id 有效，微信家的服务器将会请求网易家的服务器的接口，并且把扫描该二维码的微信用户的部分信息，如 app-id、微信头像 传递给网易，当网易服务器收到微信服务器的请求后，把微信服务器传递过来的微信用户信息保存到 session 或者其他数据缓存媒介中，此时扫描该二维码的微信用户就处于登录状态；
  5. 在上面过程进行的同时，浏览器那个请求扫描状态的接口一直在询问服务器，当上面第 4 步完成后的下一次询问，此时服务器上的该 app-id 已经有了对应的登录信息，此时接口就会返回这些登录信息；前端拿到这些信息后，把对应的页面渲染成登录后的状态即可

权限验证

原理：导航守卫 + 条件渲染

  • 页面级的权限验证：
    Vue 中结合全局的导航守卫和路由元信息（router 中的 meta 属性）来判断（比较粗犷），有权限就放行（next()），没有权限就直接拦截下来（next(false)），在页面中的 router-link 中加一个 v-if条件渲染（根据用户权限做判断，有权限的出现，没有去权限的就消失）

    如果精细一点的话使用独享的导航守卫，将每一个都细化

权限的实现：

  1. 获取权限，将权限保存到 store 和 ls 中
  2. 用权限的时候
    1. 如果是页面权限：使用导航守卫（如果是 Vue 使用 meta（路由元）保存权限信息，在导航守卫中验证；如果是 React 需要使用 Protected.js 来验证）
    2. 如果是页面元素的权限：需要根据 store 中的权限做条件渲染（Vue 中使用 v-if；如果是 React 中使用条件渲染（条件语句/表达式））

Vue 的事件元：https://router.vuejs.org/zh/guide/advanced/meta.html
