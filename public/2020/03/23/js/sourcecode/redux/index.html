state 的拆分和合并

上节将 reducer 按组件维度拆分了, 通过 combineReducers 合并了起来, 但 state 还是写在一起, 会造成 state 树很庞大、不直观、难以维护; 同理, state 也需要拆分, 一个 state, 一个 reducer 写一块

修改 createStore 函数, 增加一行 dispatch({ type: Symbol() })

思考一下这行带来的效果

  1. createStore 的时候, 用一个不匹配任何 type 的 action, 来触发 state = reducer(state, action)
  2. 因为 action.type 不匹配, 每个子 reducer 都会进到 default 项, 返回自己初始化的 state, 这样就获得了初始化的 state 树了

纯函数

什么是纯函数?

纯函数就是一种相同的输入, 永远会得到相同的输出, 而且没有任何可观察的副作用的函数

通俗来讲, 有两个要素

  1. 相同的输入, 一定会得到相同的输出
  2. 不会有 “触发事件”、更改输入参数、依赖外部参数、打印 log 等等副作用

reducer 计划函数, 必须是一个纯函数

只要传入参数相同, 返回计算得到的下一个 state 就一定相同; 没有特殊情况、没有副作用、没有 API 请求、没有变量修改, 单纯执行计算

总结

redux 中的关键名词

  • createStore

    创建 store 对象, 包含 dispatch、getState、subscribe、replaceReducer

  • reducer

    计划函数, 接受旧的 state 和 action, 生成新的 state

  • action

    一个对象, 必须包含 type 字段

  • dispatch

    dispatch(action) 触发 action, 生成新的 state

  • subscribe

    实现订阅功能, 每次出发 dispatch 时, 会执行订阅函数

  • combineReducers

    多个 reducer 的合并

  • replaceReducer

    替换 reducer 函数

  • middleware

    扩展 dispatch 函数
