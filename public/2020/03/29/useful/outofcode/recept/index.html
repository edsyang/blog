Recept
发表于 2020-03-29| 更新于 2020-05-10
字数总计:1k|阅读时长: 3 分钟
| 阅读量:

  • 主动看待问题, 想着项目因为有你这个前端而有所不同

  • 主动性和影响力, 主动性推动影响力, 而影响力象征着不会去被动接收

  • 业务和思想层次, 是否能去找对应的人提一些想法、构思

  • 跨团队合作, 一定要有一个关键的「吹鼓手」来推动项目进展, 而对业务而言, 谁最痛, 谁就承担这个角色, 推动整体大局前进, 拿到理想的结果

  • 如果你和 Leader 发生分歧: 拿出你的数据/理由/态度尽最大的努力去说服他, 如果充分沟通还行不成共识, 先听他的; 这不是媚上, 而是“在必须达成一致的前提下, 相信更有概率做出正确决定的人”

  • 会叫故事很重要, 好的故事能够启迪人, 能够有场景和代入感, 比干巴巴地谈大道理舒服得多

  • 不给自己设限, 心有多宽, 舞台就有多大; 给自己设限, say no 的时候, 就真的结束了

  • 努力选择做抬升面的点: 不做下降面的事情, 而是应该去寻找新的抬升面投入研究, 否则学术可能终生无成, 选择业务和领域也是如此

  • 真正的「组件」, 其实不应该仅限于 UI 唯独, 而应该连接接口、服务、业务规则、业务流程… 这些其实是领域模型 <= 映射 => 组件的模式, UI 万变不离其宗, 而所谓的「宗」, 就是作为的「模型」

  • 做技术产品, 要关注和同类技术的「技术壁垒」和「差异性」打造, 不做无差异性的东西

  • 横向的东西, 懂得借力, 越是基础的东西, 越应该用好内部已有的资源体系, 个人的发力点应该基于业务做「高层次的设计」

  • 共建的基础是, 都有同类的痛点和诉求, 且优先级都比较靠前, 不要为了共建而共建; 而对于友方团队, 保证底层技术和标准的「互通」, 未来才有业务交叠、共建的机会

  • 回击质疑, 最好的方法是用结果超越别人

  • 这个事情你做, 就不允许别人做, 这是最大的破坏, 良性的竞争则是要促进团队快速往前发展

  • 创新是真正提供有业务价值、有技术价值的方案

  • 战略阶段

    • 暴漏目标: 讲自己的方案
    • 统一思想: 尽量统一思想
    • 调整组织: 因为统一思想的时候很多人充数, 当真正落实的时候这些才会真正暴露出来

  • 在做客服工作时, 一定要尽可能知晓用户背景, 要理解用户遇到问题时的情绪, 以及要洞悉用户使用产品时的心智模型

  • 理解用户的心智, 就是搞清楚用户究竟想要什么, 这里的难点是如何知道自己的理解, 不是少部分用户的心智, 而是大部分用户的心智; 究竟谁理解的能代表大部分用户? 这个问题除了从数据取证, 还需要拼谁「傻」, 设能瞬间傻乎乎, 谁就能代表用户; 这不代表用户傻, 而是让自己变成傻乎乎的状态后, 从傻的方向往聪明的方向看, 更能看到并看透聪明, 从聪明的方向往傻看, 往往会心生优越或鄙夷, 从而看不到看不清

  • 理解用户心智的关键点, 不是懂道理, 而是疯狂去实践, 刻意让自己变傻, 不断练习, 不断掏空自己知道的, 不断变化角度去看世界, 不断去想用户在想什么, 直到自己随时随刻能自然变傻
