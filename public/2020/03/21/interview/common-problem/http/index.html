浏览器在输入 url 之后点击回车之后浏览器做了什么

  • 请求阶段:
    1. 输入地址, 按下 enter 键;
    2. 将网站发送给 dns 服务器, dns 服务器会将网址解析成 ip 地址
    3. 把请求内容发送到 ip 地址对应的服务器
  • 响应阶段
    • (服务器)分析请求内容, 整理需要的内容, 将内容通过 http 协议返回(后端的响应中会告诉前端相应内容类型, 在响应头中有一个 Context-Type 属性, 表示相应内容类型, 目的是告诉前端按不同类型去解析返回的响应内容)
  • 渲染阶段:
    1. 解析 html 文档, 形成 DOM Tree, 遇到 link、img、script、audio、video 会再次发送 http 请求对应资源
    2. 解析 CSS 文档, 形成 CSSOM Tree
    3. 将 DOM Tree、CSSOM Tree 组合为 Render Tree, 并将其交给 GPU 渲染

对 http 的理解

  1. HTTP协议是无状态的
    • 就是说每次HTTP请求都是独立的，任何两个请求之间没有什么必然的联系。但是在实际应用当中并不是完全这样的，引入了Cookie和Session机制来关联请求
  2. 多次HTTP请求
    • 在客户端请求网页时多数情况下并不是一次请求就能成功的，服务端首先是响应HTML页面，然后浏览器收到响应之后发现HTML页面还引用了其他的资源，例如，CSS，JS文件，图片等等，还会自动发送HTTP请求这些需要的资源。现在的HTTP版本支持管道机制，可以同时请求和响应多个请求，大大提高了效率
  3. 基于TCP协议
    • HTTP协议目的是规定客户端和服务端数据传输的格式和数据交互行为，并不负责数据传输的细节。底层是基于TCP实现的。现在使用的版本当中是默认持久连接的，也就是多次HTTP请求使用一个TCP连接

http 缓存机制

http 缓存

  1. http 报文(通信时用于承载交互信息的载体)
    • 报文首部
    • 空行（CR+LF）：Carriage Return 回车符 + Line Feeding
    • 报文主题
  2. request 报文
    • 报文首部: 请求行(请求方式、版本协议、url) 和请求头(通用、实体、其它)
    • 空行
    • 请求体: 只有 POST 才会有请求体, GET 会自动忽略
  3. response 报文
    • 报文首部: 状态行(http协议版本、响应状态码（http status）、响应码对应短语) 和响应头
    • 空行
    • 响应体: 服务端响应的主要内容

缓存机制:

浏览器存在一个缓存数据库, 用于存储缓存信息; 当第一次加载的时候, 缓存数据库中没有对应的数据, 所以需要请求服务器, 服务器将数据返回后, 会将数据存储至缓存数据库中

缓存规则:

  1. 强制缓存: 先去缓存数据库中查找, 如果生效, 不需要再和服务器发生交互
    • 缓存规则信息包含在响应 header 中
    • 响应 header 中会有两个字段来标明失效规则（Expires 到期时间 / Cache-Control 规则）
  2. 对比缓存: 无论是否生效, 都需要与服务器发生交互

    强制缓存的优先级高于对比缓存, 所以会先执行强制缓存, 如果没有的话在执行对比缓存

一个 tcp 链接能发几个 http 请求

  • http 1.0
    • 一般情况不支持长连接, 所以在每次发送完毕后, tcp 连接会自动断开, 因此 tcp 发送一个 http 请求
    • 但是通过 Connection 和 Keep-Alive 首部来实现一条 tcp 连接保持在活跃状态, 在请求头上带上 Connection: Keep-Alive, 并且通过 Keep-Alive 通用首部中指定的, 用逗号分隔的选项调节 keep-alive 的行为; 如果客户端和服务端都支持, 可以发送多条, 但是这个方式也有限制(参 Keep-Alive 连接的限制和规则)
  • http 1.1
    • 支持长连接, 只要 tcp 连接不断开, 便可以一直发送 http 请求, 且没有上限
  • http 2.0
    • 支持多用复用, 一个 tcp 连接是可以并发多个 http 请求的
    • 支持长连接, 只要 tcp 连接不断开, 便可以一直发送 http 请求, 且没有上限

对 cookie & session & token 的理解

产生背景、原理、有什么问题
chorme80 屏蔽第三方 cookie 事件

区别:

  1. session 是服务端技术，session 存在服务器上
  2. cookie 是 http 协议的一部分，存在客户端
  3. token

联系: session-id 存在 cookie 中，会随着 http 通信时在客户端和服务端来回传递

cookie & localStorage & sessionStorage 的区别

  • cookie
  • localStorage: 本地永久存储, 只能手动或者代码删除
  • sessionStorage: 会话存储, 页面关闭就没有了
特性 cookie localStorage sessionStorage
数据的生命期 一般由服务器生成，可设置失效时间。如果在浏览器端生成Cookie，默认是关闭浏览器后失效 除非被清除，否则永久保存 仅在当前会话下有效，关闭页面或浏览器后被清除
存放数据大小 4K左右 一般为5MB
与服务器端通信 每次都会携带在HTTP头中，如果使用cookie保存过多数据会带来性能问题 仅在客户端（即浏览器）中保存，不参与和服务器的通信
易用性 需要程序员自己封装，源生的Cookie接口不友好 源生接口可以接受，亦可再次封装来对Object和Array有更好的支持

get / post 的区别

  • get: 一般用于从服务器获取数据，但也可以向服务器传递/发送少量的数据

    1. GET请求通过url末尾以问号传参的形式向服务器传递数据，如?key1=value1&key2=value2（通过location.search 获取地址栏里面url的上的问号传参）；

      • 向url末尾拼接参数（这些参数也叫查询字符串），如果原有的url有?就不需要再写?，直接用&符和之前的分隔

      • 这些参数以key=value的形式存在，key一般都是服务端定好的，服务端要求传什么就传什么

    2. 因为GET发送数据给服务端，都是把数据拼接在url末尾，但是url的总长度是有限的；一般IE是2k，Chrome是8k；所以GET请求能传递/发送给服务器的数据量很小

    3. GET请求传递数据是以明文的形式传递，不安全

    4. GET请求容易走缓存，一般情况下访问接口的数据不需要缓存；所以一般情况下在url末尾拼接随机数或者时间戳来清除缓存（原理：浏览器检测到url发生变化，就不会用缓存的内容，而是重新请求数据）

  • post: 一般用于把数据传递给服务器或需要传递大量数据

    1. post请求把数据放到请求体中传递给服务端（字符串类型的）
    2. 发送给服务器的数据是放到请求体中的，所以传递数据的大小没有限制
    3. post请求是把数据放到请求体里面，相对来说安全一点
    4. post请求不会走缓存
