react / vue (mvc / mvvm) 的区别

mvc/mvvm 区别

  1. 设计思想不同
  2. 对数据变换的感知不同
  • react
    • 单向数据流
    • 推崇结合 immutable 来实现不可变数据
    • react 设计了 jsx(createElement 的语法糖), 还是通过 js 操作
    • 可通过 hoc 来扩展
    • 很多事情交给社区, 拓展性强
  • vue
    • 双向数据流(响应式)
    • 基于的是数据可变, 通过对每一个属性简历 watcher 来监听
    • vue 是把 html、css、js 组合到一起
    • 通过 mixins 来扩展
    • 很多事情已经内置, 使用方便

react 性能优化是需要手动操作, 而 vue 是自动的, 但是当 state 特别多的时候, watcher 也会特别多, 所以当项目中状态特别多的时候, 使用 react 更加可控

  • mvc - 数据更改视图跟着更改
    • model 模型: 处理应用程序数据逻辑部分
    • view 视图: 处理应用程序数据显示部分
    • controller 控制器: 处理应用程序中用户交互的部分
  • mvvm - 数据更改视图跟着更改，视图更改数据也跟着更改
    • model
    • view
    • view-model

MVVM 降低了 controller 的存在感, 正因为如此, 有些数据变得不可控

react 原理(dom-diff)

总结:

  1. 把树形结构按照层级分解，只比较同级元素
  2. 列表结构的每个单元添加唯一的 key 属性，方便比较
  3. React 只会匹配相同 class 的 component（这里面的 class 指的是组件的名字）
  4. 合并操作，调用 component 的 setState 方法的时候, React 将其标记为 dirty.到每一个事件循环结束, React 检查所有标记 dirty 的 component 重新绘制
  5. 选择性子树渲染。开发人员可以重写 shouldComponentUpdate 提高 diff 的性能

传统的 diff 算法复杂度过高, react 优化了 diff 算法的复杂度

diff 策略

  1. Web UI 中 DOM 节点跨层级的移动操作特别少，可以忽略不计
  2. 拥有相同类的两个组件将会生成相似的树形结构，拥有不同类的两个组件将会生成不同的树形结构
  3. 对于同一层级的一组子节点，它们可以通过唯一 id 进行区分

基于以上策略, react 对 tree diff、component diff、element diff 进行了优化

  • tree diff
    • react 只会对相同层级的 dom 节点进行比较, 即同一个父节点下所有的子节点
  • component diff
    • 如果是同一类型的组件，按照原策略继续比较 Virtual DOM 树即可
    • 如果不是，则将该组件判断为 dirty component，从而替换整个组件下的所有子节点
    • 对于同一类型的组件，有可能其 Virtual DOM 没有任何变化，如果能够确切知道这点，那么就可以节省大量的 diff 运算时间。因此，React允许用户通过shouldComponentUpdate()来判断该组件是否需要进行diff算法分析
  • element diff
    • 当节点处于同一层级时，diff 提供了 3 种节点操作，分别为 INSERT_MARKUP (插入)、MOVE_EXISTING (移动)和 REMOVE_NODE (删除)
    • INSERT_MARKUP :新的组件类型不在旧集合里，即全新的节点，需要对新节点执行插入操作
    • MOVE_EXISTING :旧集合中有新组件类型，且 element 是可更新的类型，generateComponentChildren 已调用receiveComponent ，这种情况下 prevChild=nextChild ，就需要做移动操作，可以复用以前的 DOM 节点
    • REMOVE_NODE :旧组件类型，在新集合里也有，但对应的 element 不同则不能直接复用和更新，需要执行删除操作，或者旧组件不在新集合里的，也需要执行删除操作

react 虚拟 dom

概念: 用新渲染的对象树去和旧的树进行对比，记录这两棵树差异。记录下来的不同就是我们需要对页面真正的 DOM 操作，然后把它们应用在真正的 DOM 树上，页面就变更了

步骤:

  1. 用 JavaScript 对象结构表示 DOM 树的结构；然后用这个树构建一个真正的 DOM 树，插到文档当中
  2. 当状态变更的时候，重新构造一棵新的对象树。然后用新的树和旧的树进行比较，记录两棵树差异
  3. 把2所记录的差异应用到步骤1所构建的真正的DOM树上，视图就更新了

Virtual DOM 本质上就是在 JS 和 DOM 之间做了一个

react 生命周期(按阶段来说)

  • 首次渲染阶段: constructor - componentWillMount - render - componentDidMount
  • 状态更新触发: shouldComponentWillUpdate - componentWillUpdate - render - componentDidUpdate
  • 属性更新触发: componentWillReceiveProps - shouldComponentWillUpdate - componentWillUpdate - render - componentDidUpdate
  • 卸载前会触发: componentWillUnmount

请求数据在哪个生命周期?为什么?

一般会在 componetDidMount 中调用, 原因: componetDidMount 是在组件已经完全挂载到网页上才会被调用执行, 所以可以保证数据的加载、保证组件正确渲染, 而且在这个方法中调用 setState, 会触发重新渲染, 官方设计这个方法就是用来加载外部数据或处理其他副作用代码使用, 其次 componentWillMount 会导致请求执行两次, 一次在服务端一次在客户端, 另外如果在这里请求异步数据的话, 数据一定会在 render 后才到, 如果没有设置初始状态, 用户体验会不好, 所以 componetWillMount 才会即将废弃, 需要加一个 UNSAFE_ 前缀

调用 setState 都做了哪些工作

  • setState 的同异步问题, 把 setState 看作是一个请求而不是一个立即执行的命令, 为了更好的性能, react 可能延迟它, 把若干个更新放到一起执行, 所以在同步执行中它是异步的, 反之则是同步的
  • 在调用 setState 时, setState 会把 state 的变化插入队列, 告诉 react 这个组件及它的子组件需要根据更新后的 state 重新渲染, react 得到元素树之后, 会自动计算新老树节点差异, 然后根据差异对界面进行最小化重新渲染(差异算法中, react 会相对精确的知道哪些位置发生变化以及如何变化, 这就保证了按需更新, 而不是全部重新渲染)

react 16.8 新增的东西

最主要的就是 react-hooks

react 性能优化

  1. immutable -> seamless-immutable -> immer.js
    • Map: 键值对集合, 对应 Object, Es6 种也有专门的Map对象
    • List: 有序可重复列表, 对应于 Array
    • ArraySet: 有序且不可重复的列表
  2. pureComponent 实现 shouldComponentUpdate 生命周期的方法
  3. 减少新建变量和 bind 函数

react 按需加载

  • react.lazy + <Suspense>
  • 其他的第三方库也可以实现按需加载的功能

react-router 原理

实现 URL 与 UI 界面的同步, 其中在 react-router 中, URL 对应 location 对象, 而 UI 是由 react 的 components 来决定的，这样就转变成 location 与 components 之间的同步问题

优点:

  • 与 React 融为一体, 专为 react 量身打造, 编码风格与 react 保持一致, 例如路由的配置可以通过 component 来实现
  • 不需要手工维护路由 state，使代码变得简单
  • 强大的路由管理机制，体现在如下方面
    • 路由配置: 可以通过组件、配置对象来进行路由的配置
    • 路由切换: 可以通过 <Link> Redirect 进行路由的切换
    • 路由加载: 可以同步记载, 也可以异步加载, 这样就可以实现按需加载
  • 使用方式: 不仅可以在浏览器端的使用, 而且可以在服务器端的使用

HashRouter 和 BrowserRouter 的区别

  • HashRouter: 使用 URL 的哈希部分（即 window.location.hash）来保持页面的 UI 与 URL 同步, hashRouter 不支持location.key、location.state
  • BrowserRouter: 使用 HTML5 历史 API（ pushState、replaceState 和 popstate 事件）, 让页面的 UI 与 URL 同步

简单说一下 redux

进行状态统一管理的类库, 多个组件之间想要实现信息的共享都可以通过 redux 解决, 把共享信息存储到 redux 中进行管理, 还可以做性能优化, 使用 redux 进行临时存储

工作流:

  1. createStore 创建存储状态的容器, 容器中的状态信息会放到 subscribe 的事件池中, 当其中的状态改变会通知事件池中的方法执行
  2. 当一个组件想获取 redux 中的状态信息, 调用 getState
  3. 当一个组件想修改 redux 中的状态信息, 调用 dispatch, 派发一个行为标示, 通知 reducer(状态管理员) 中对应的修改状态任务执行, reducer 变更后将数据传回容器中
  4. 组件根据 redux 中的状态变化, 需要 subscribe, 当更新后会自动重新渲染
