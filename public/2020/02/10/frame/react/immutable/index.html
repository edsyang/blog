shouldComponentUpdate

shouldComponentUpdate(nextProps, nextState)

  • React 中的生命周期, 在渲染执行之前被调用(首次渲染/使用 forceUpdate() 时不会调用)
  • 默认行为是 state 每次发生变化组件都会重新渲染
  • 可以根据这个生命周期来判断组件中当前的和即将收到的 state、props 来决定是否重新渲染该组件
  • 这个生命周期可以且仅可以作为性能优化的方式而存在

注意

  1. 返回 false 后, 不会调用 UNSAFE_componentWillUpdate()、render()、componentDidUpdate(), 并且不会阻止子组件在 state 更改时重新渲染
  2. 不要在该生命周期中进行深层比较或者使用 JSON.stringifg(), 会影响效率、损害性能
  3. 如果想要阻止渲染, 应该使用内置的 PureComponent 组件, 而不是使用这个(可能会产生 bug)

React.PureComponent

React.Component 和 React.PureComponent 很相似, 区别在于前者未实现 shouldComponentUpdate(), 而后者以浅层对比 props 和 state 的方式实现了 shouldComponentUpdate()

注意

  1. React.PureComponent 中的 shouldComponentUpdate() 仅作为对象的浅层比较, 所以只有 props 和 state 较为简单的时候才可以使用 React.PureComponent
  2. React.PureComponent 中的 shouldComponentUpdate() 会跳过所有子组件树的 props 更新, 所以需要保证它下面的所有子组件都是“纯”组件
  3. 如果对象中包含复杂的数据结构, 则可能没有办法检查深层的差别从而导致错误的对比结果
  4. 如果在深层数据结构发生变化时调用 foceUpdate() 来确保组件被正确更新或者使用 immutable.js 加速嵌套数据的比较

React.memo

因为函数式组件没有 shouldComponentUpdate 生命周期, 所以 PureComponent 只适用于 class 类声明组件; 那么, 在函数式组件中, 如果要做像 PureComponent 一样的事情, 就需要使用 React.memo (相当于函数组件中的 PureComponent)

注意

  1. React.memo 为高阶函数, 只适用于函数式组件
  2. React.memo 仅影响 props 变更, 如果里面使用了 useState 或 useContext, 那么当 context 发生变化时, 仍会重新渲染
  3. 使用 React.memo 包裹后, 在 props 不变的情况下, 会跳过渲染组件的操作并直接复用最近一次渲染的结果
  4. 和 PureComponent 一样, 该方法仅作为性能优化的方法, 不要在依赖它阻止渲染, 可能会出现 bug
  5. 和 PureComponent 一样, 只会对数据结构进行浅层对比; 如果想要控制对比过程, 将自定义的比较函数通过第二个参数传入来实现

immutable.js

由 Facebook 研发, immutable.js 通过结构共享提供了不可变、持久化集合, 并拥有以下特性

  1. 不可变: immutable Date 一旦创建, 一个集合便不能再被修改, 对 immutable 对象的任何操作都会返回一个新的 immutable 对象
  2. 持久化数据结构(Persistent Data Structure): 使用旧数据创建新数据的同时保证旧数据可用且不可变
  3. 结构共享(Structural Sharing): 如果对象树中一个节点发生变化, 只修改这个节点和它受影响的父节点, 其他节点则进行共享; 避免深克隆把所有节点都复制一遍带来的性能损耗

优点

  1. 降低可变数据带来的复杂度; 可变数据耦合了 Time 和 Value 的概念, 造成了数据很难被回溯
  2. 结构共享会尽量复用内存, 没被饮用的对象会被垃圾回收
  3. 方便实现 Undo/Redo、Copy/Paste 以及时间旅行的功能; 因为每次的数据都不一样, 所以只要把这些数据放到一个数组中存储起来, 就方便做这些操作了
  4. 并发安全
  5. 拥抱函数式编程; 纯函数式编程, 只要输入一致, 输出必然一致, 有利于开发的调试和组装

缺点

  1. 学习成本
  2. 增加资源文件大小
  3. 容易和原生对象混淆

    使用以下做法可以尽量避免

    • 使用 Flow 或 TypeScript 静态类型检查工具
    • 约定变量名规则, 比如 immutable 类型都以 _ 或 $$ 开头
    • 使用 immutable.fromJS 而不是 immutable.Map 或 immutable.List 来创建对象, 这样可以避免 immutable 和原生对象间的混淆

其中最重要的三种数据结构

  • Map: 键值对集合, 对应与 Object, ES6 也有专门的 Map 对象
  • List: 有序可重复的列表, 对应于 Array
  • Set: 无序且不可重复的列表

这样的机制十分符合 PureComponent 浅层比较的特性 —— 只在内容改变时重新渲染; 所以使用 Immutable 和 PureComponent 可以实现性能提升

seamless-immutable.js

与 immutable 相似, 只不过它并没有实现完成的持久化数据结构, 而是使用 Object.defineProperty 扩展了 JavaScript 的 Array 和 Object 对象来实现, 只支持 Array 和 Object 两种数据结构, API 基于 Array 和 Object 操作不变; 代码库很小, 轻量, 压缩后只有 2k, 而 immutable 压缩后有 16k

Immer

由 Mobx 的作者研发, 主要解决利用元编程化 immutable 使用的复杂度; 是一个支持柯里化、仅支持同步计算的工具; 适用于作为 redux 的 reducer 使用

这就是 immer: Create the next immutable state by mutating the current one.(通过改变当前状态来创建下一个不可变状态)
