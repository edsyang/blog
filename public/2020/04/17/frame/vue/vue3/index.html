前言

vue-next/reactivity

Vue3 中, ref 和 reactive 是整个源码中的核心, 通过这两个方法创建了响应式数据, 也是吃透 reactivity 的基础, 在这之前, 首先要阅读一下 Composition API 的介绍, 会更快的理解源码以及更好的认识这个库

Ref vs Reactive

Ref 最重要的作用就是提供了一套 Ref 类型

  • TS 中 is 关键词
  • UnwrapNestedRefs & UnwrapRef 的了解建立在 TS 的 infer

Ref 有个 key 为 Symbol 的属性做类型标识, 有个 value 用来存储数据; 这个数据可以是任意类型, 但不能是被嵌套 Ref 类型的类型, 具体来讲就是不能是 Array<Ref> 或者 { [key]: Ref }; 但是 Ref<Ref> 就可以..

TS 最新版, 支持递归类型, 如果使用最新版 TS 来递归解套的话, 可以这样实现

Ref 具体体现

isObject 中 为什么不是 val is object?

这个 ref 函数, 定义了 get/set, 但没有任何 Proxy 相关的操作, 而 reactive 能构建出响应式数据的前提是传参必须是对象, 但 ref 入参是对象时, 同样也需要 reactive 做转化; 在 Ref vs Reactive 中进行了说明

However, the problem with going reactive-only is that the consumer of a composition function must keep the reference to the returned object at all times in order to retain reactivity. The object cannot be destructured or spread

对于基本数据类型, 函数传递或对象解构时, 会丢失原始数据的引用, 换句话讲, 就是没法让基本数据类型, 或者解构后的变量(如果它的值是基本数据类型的话), 成为响应式的数据

但有的时候, 确实就需要一个数字/字符串是响应式的, 或者就是想利用解构的写法; 那只能通过创建一个对象, 也就是 Ref 数据, 将原始数据保存在 Ref 的属性 value 当中, 再将它的引用返回给使用者, 因为是自己创造出来的对象, 所以也没必要使用 Proxy 做代理, 直接劫持这个 value 的 get/set 即可, 这就是 ref 函数和 Ref 类型的由来

不过单靠 ref 还没发解决对象解构问题, 它只是将基本数据保持在一个对象的 value 中, 以实现数据响应式; 对于对象的解构, 还需要另外一个函数: toRefs

通过遍历对象, 将每个属性值都转成 Ref 数据, 这样解构出来的还是 Ref 数据, 自然就保持了响应式数据的引用;

注意: toRefs 函数中引用的是 toProxyRef 而不是 ref, 它并不会在 get/set 中注入 trace 和 trigger, 也就是说, 向 toRefs 传入一个正常的对象, 是不会返回一个响应式数据的, 必须要传递一个已经被 reactive 执行返回的对象才能有响应式的效果,
